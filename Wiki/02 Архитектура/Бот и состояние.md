# Бот и состояние

## Подтверждённые факты

- Точка входа: `asyncio.run(run())` в [bot.py](C:\Users\sergk\OneDrive\Desktop\traffichubserver\content-poster\bot.py).
- Runtime-модель: один Python-процесс с `python-telegram-bot`, long polling через `app.updater.start_polling()`.
- Админ-панель реализована внутри Telegram-диалога, а не как отдельный web/frontend сервис.
- AI-генерация ходит во внешний OpenRouter API через `aiohttp`.

## Файлы состояния

- `posts.json` — очередь/история постов.
- `users.json` — баланс и счётчики генераций пользователей.
- `billing_config.json` — стоимость одной генерации.
- `system_prompt.txt` — шаблон промта для OpenRouter.
- `vacancy_config.json` — конфиг типов вакансий и ссылок.

## Почему так

Проект сделан как single-file bot с file-based persistence. Это упрощает запуск без БД, но повышает риск runtime-ошибок при несинхронных изменениях в коде и state-файлах.

## Важные runtime-нюансы

- `menu_fallback` должен возвращать реальное состояние conversation, если он запускает новый сценарий (`✏️ Создать пост`), иначе следующий ввод потеряется.
- `users.json` не обязан быть предзагруженным вручную: админ-цепочка может создать пользователя по числовому Telegram ID на лету.
- `load_system_prompt()` имеет встроенный fallback на дефолтный шаблон, поэтому отсутствие `system_prompt.txt` не должно ломать AI-цепочку.

## Ограничения и риск

- Нет транзакционности и блокировок на файловом слое.
- Нет отдельного error handler для `python-telegram-bot`; ошибки уходят в `bot.log`.
- Проверка “полностью работает” требует реального Telegram runtime, одной компиляции недостаточно.
