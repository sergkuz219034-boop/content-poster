# Гипотеза

`menu_fallback` завершал conversation независимо от того, какой сценарий он запускал, а balance-flow падал на отсутствующих user-records вместо их создания.

# Проверка

- `bot.py` показал, что `menu_fallback` возвращал `ConversationHandler.END` даже после вызова `create_post_start(...)`.
- В `finish_post_query(...)` был найден прямой runtime-баг с `update.effective_user.id` в callback-контексте.
- Админский шаг изменения баланса принимал только уже существующие записи в `users.json`.
- После правки `python -m py_compile bot.py` проходит без ошибок.

# Наблюдение

- `users.json` теперь может создаваться лениво по числовому Telegram ID.
- Переход из reply-keyboard меню в новый сценарий должен возвращать реальное состояние conversation, а не всегда `END`.

# Вывод

Это была связка из трёх хрупких мест: неверный callback-контекст, слишком жёсткий fallback-роутинг и зависимость админки от предзагруженных user-records. После фикса цепочка стала устойчивее.
